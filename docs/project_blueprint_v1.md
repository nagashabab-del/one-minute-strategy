# المخطط التشغيلي لرحلة المشروع

## بيانات الوثيقة
- الإصدار: `v1`
- التاريخ (UTC): `2026-03-09T00:40:20Z`
- الهدف: تحويل المنتج إلى منصة إدارة مشاريع متسقة من بداية المشروع حتى الإغلاق.

## تعريف المنتج
المنتج هو منصة إدارة مشاريع.

المستشارون الذكيون جزء من مرحلة تحليل القرار داخل الرحلة، وليسوا المنتج كاملًا.

## المبدأ المعماري
1. رحلة مشروع واحدة واضحة.
2. كل مرحلة لها شروط دخول وخروج صريحة.
3. لا انتقال بين المراحل بدون اكتمال الحد الأدنى من البيانات.
4. لا تكرار للرسائل في أكثر من شاشة؛ الرسالة تظهر في نقطة واحدة مع إجراء واضح.

## رحلة المشروع المعتمدة
1. تسجيل المشروع والبيانات الأساسية.
2. جلسة النقاش مع المستشارين.
3. التوصيات والقرار التنفيذي.
4. التخطيط التنفيذي.
5. المتابعة التشغيلية.
6. التقارير.
7. إغلاق المشروع.

## المرحلة 1: تسجيل المشروع والبيانات الأساسية
### الهدف
تأسيس مشروع قابل للتحليل والتنفيذ.

### البيانات الإلزامية
1. اسم المشروع.
2. نوع المشروع.
3. تاريخ البداية.
4. تاريخ النهاية.
5. الميزانية التقديرية.
6. نوع الموقع.

### سلوك الواجهة
1. عرض نموذج قصير وواضح.
2. إظهار نسبة اكتمال البيانات.
3. زر رئيسي واحد:
`حفظ ومتابعة`.

### المخرجات
1. سجل مشروع نشط.
2. حالة مشروع: `جاهز للنقاش`.

### شرط الانتقال
اكتمال الحقول الإلزامية الستة.

### حالات التعثر
1. نقص بيانات: منع الانتقال مع إظهار الحقول الناقصة.
2. تاريخ غير منطقي: منع الحفظ برسالة واضحة.

## المرحلة 2: جلسة النقاش مع المستشارين
### الهدف
جمع أسئلة وإجابات تولّد أساسًا تحليليًا موثوقًا.

### البيانات الإلزامية للدخول
حالة المشروع `جاهز للنقاش`.

### سلوك الواجهة
1. عرض المستشارين المختارين.
2. عرض الأسئلة بشكل تدريجي.
3. حفظ تلقائي بعد كل إجابة.

### المخرجات
1. سجل أسئلة مكتمل.
2. سجل إجابات مكتمل.
3. حالة مشروع: `جاهز للتوصيات`.

### شرط الانتقال
اكتمال الحد الأدنى من الإجابات المطلوبة.

### حالات التعثر
1. انقطاع الجلسة: استعادة آخر نقطة.
2. نقص إجابات: إظهار ما تبقى فقط.

## المرحلة 3: التوصيات والقرار التنفيذي
### الهدف
تحويل الحوار إلى قرار قابل للتنفيذ.

### سلوك الواجهة
1. ملخص توصيات موحد.
2. نقاط قوة وفجوات ومخاطر.
3. قرار نهائي بصيغة واضحة:
`اعتماد`
أو
`إعادة التحليل`.

### المخرجات
1. ملخص قرار.
2. حالة مشروع: `قرار معتمد`.

### شرط الانتقال
وجود قرار معتمد.

### حالات التعثر
1. تضارب توصيات: طلب جولة إيضاح إضافية.
2. غموض القرار: منع الانتقال حتى اعتماد نهائي.

## المرحلة 4: التخطيط التنفيذي
### الهدف
تفكيك القرار إلى خطة تشغيل.

### الوحدات
1. النطاق.
2. الميزانية.
3. الخطة الزمنية.
4. المخاطر.

### سلوك الواجهة
1. كل وحدة في صفحة مستقلة.
2. انتقال تسلسلي واضح.
3. شريط تقدم موحد.

### المخرجات
1. نطاق معتمد.
2. ميزانية أساسية.
3. خطة زمنية مع تبعيات.
4. سجل مخاطر أولي.
5. حالة مشروع: `جاهز للمتابعة`.

### شرط الانتقال
اكتمال الوحدات الأربع.

### حالات التعثر
1. ميزانية غير متوازنة: تنبيه إلزامي قبل المتابعة.
2. خطة بلا تبعيات: تنبيه جودة قبل الاعتماد.

## المرحلة 5: المتابعة التشغيلية
### الهدف
تحويل الخطة إلى تنفيذ يومي مضبوط.

### سلوك الواجهة
1. لوحة مهام يومية.
2. حالات المهام: لم يبدأ، جاري، متعثر، مكتمل.
3. تنبيهات انحراف وقت وتكلفة.

### المخرجات
1. تقدم فعلي قابل للقياس.
2. قرارات تصحيحية موثقة.
3. حالة مشروع: `جاهز للتقرير`.

### شرط الانتقال
إغلاق المهام الحرجة أو تبريرها إداريًا.

## المرحلة 6: التقارير
### الهدف
إنتاج مخرجات تنفيذية للإدارة.

### سلوك الواجهة
1. قائمة تقارير مرتبطة بالمشروع.
2. حالات التقرير: مسودة، مكتمل، معتمد.
3. فتح التقرير وتتبعه حسب المعرّف.

### المخرجات
1. تقرير تنفيذي معتمد.
2. حالة مشروع: `جاهز للإغلاق`.

### شرط الانتقال
وجود تقرير نهائي معتمد.

### حالات التعثر
1. تقرير غير موجود: يجب أن يعيد حالة صفحة غير موجودة صحيحة.
2. تقرير غير معتمد: منع الإغلاق.

## المرحلة 7: إغلاق المشروع
### الهدف
إقفال المشروع رسميًا مع حفظ المعرفة.

### سلوك الواجهة
1. قائمة تحقق قبل الإغلاق.
2. ملخص نهائي.
3. أرشفة قابلة للرجوع.

### المخرجات
1. مشروع مغلق.
2. أرشيف قابل للبحث.
3. سجل دروس مستفادة.

## الأدوار والصلاحيات
1. مدير المشروع: قيادة المراحل واعتماد الانتقال.
2. مدير العمليات: متابعة التنفيذ اليومي.
3. المدير المالي: اعتماد المسار المالي.
4. المشاهد: قراءة فقط.

## ربط المسارات الحالية بالرحلة المعتمدة
1. `/app` = مدخل القرارات والإجراء التالي.
2. `/app/strategy/brief` = تسجيل البيانات الأساسية.
3. `/app/strategy/advisors` و`/dialogue` = النقاش.
4. `/app/strategy/decision` = القرار.
5. `/app/strategy/execution/*` = التخطيط والتنفيذ.
6. `/app/workflows` = المتابعة التشغيلية.
7. `/app/reports` = التقارير.
8. `/app/settings` = الحوكمة والإعدادات.

## فجوات حالية يجب إغلاقها
1. تكرار رسائل التعثر في عدة شاشات بدل نقطة توجيه واحدة.
2. عدم اتساق بعض التسميات بين الصفحات.
3. ازدحام معرفي في صفحات تنفيذ متقدمة للمستخدم المبتدئ.
4. سلوك صفحة التقرير غير الموجود يحتاج حالة نهائية متوافقة.

## خطة التنفيذ القادمة بناء على هذا المخطط
1. اعتماد شروط الانتقال لكل مرحلة في خدمة مركزية.
2. نقل المراحل تدريجيًا إلى ملفات مستقلة.
3. إبقاء المسارات الحالية مع توافق انتقالي حتى اكتمال النقل.
4. حذف القديم فقط بعد تحقق كامل من التغطية السلوكية.

## معايير القبول قبل كل دمج
1. نجاح فحوص الجودة.
2. نجاح رحلة مستخدم كاملة من تسجيل المشروع إلى التقرير.
3. توثيق الأدلة البصرية.
4. عدم كسر البيانات الحالية أو فقدانها.

# خطة النقل المرحلي لإعادة بناء المنتج

## بيانات الوثيقة
- الإصدار: `v1`
- التاريخ (UTC): `2026-03-09T00:27:24Z`
- النطاق: إعادة تنظيم المنتج ليعمل كمنصة إدارة مشاريع كاملة دون هدم البناء الحالي.

## الهدف التنفيذي
إعادة بناء بنية المنتج حول رحلة مشروع واحدة من البداية إلى الإغلاق، مع نقل تدريجي آمن من الملفات الحالية إلى ملفات جديدة مستقلة لكل مرحلة.

## المبادئ الحاكمة
1. لا هدم كامل ولا إعادة كتابة شاملة دفعة واحدة.
2. النقل يتم مرحلة بمرحلة مع توافق تشغيلي مستمر.
3. كل مرحلة جديدة في ملف مستقل.
4. منع إضافة ميزات جديدة خارج نطاق رحلة المشروع حتى اكتمال النقل الأساسي.
5. القديم يبقى كمسار احتياطي حتى اجتياز معايير الاستبدال.

## النموذج المستهدف لرحلة المشروع
1. تسجيل المشروع والبيانات الأساسية.
2. جلسة النقاش بين مدير المشروع والمستشارين.
3. توصيات المستشارين والقرار التنفيذي.
4. التخطيط التنفيذي (النطاق، الميزانية، الخطة، المخاطر).
5. المتابعة التشغيلية والاعتمادات.
6. التقارير.
7. إغلاق المشروع.

## معمارية النقل المقترحة
### 1) بنية ملفات جديدة مستقلة
- `app/app/strategy-v2/`
- لكل مرحلة ملف صفحة مستقل ومكوّنات فرعية مستقلة.
- الخدمات المشتركة في `app/app/_lib/` لتجنب تكرار المنطق.

### 2) طبقة توافق انتقالية
- مبدّل مسار داخلي يوجّه بين القديم والجديد حسب مرحلة النقل.
- عدم كسر الروابط الحالية أثناء الانتقال.

### 3) مصادر البيانات
- الحفاظ على المصدر الحالي مؤقتًا.
- بناء طبقة قراءة/كتابة موحدة.
- أي تغيير على مفاتيح التخزين يتم عبر طبقة توافق فقط.

## مراحل التنفيذ التفصيلية
### المرحلة 0: التثبيت قبل النقل
1. تثبيت تعريف المنتج رسميًا كمنصة إدارة مشاريع.
2. تجميد تطوير الميزات الجديدة خارج الرحلة الأساسية.
3. اعتماد هذه الوثيقة كمرجع وحيد لقرارات النقل.

### المرحلة 1: إدخال المشروع
1. إنشاء صفحة مرحلة جديدة مستقلة لإدخال البيانات الأساسية.
2. تحديد الحقول الحرجة المطلوبة للانتقال.
3. ربط التوجيه إلى المرحلة التالية فقط عند اكتمال الحد الأدنى.

### المرحلة 2: النقاش مع المستشارين
1. نقل منطق جلسة النقاش إلى وحدة مستقلة.
2. فصل واجهة النقاش عن منطق التحقق من البيانات.
3. منع الوصول لهذه المرحلة عند نقص حقول المرحلة السابقة.

### المرحلة 3: التوصيات والقرار
1. نقل شاشة القرار والتوصيات كوحدة مستقلة.
2. تحديد حالة واضحة للقرار (مسودة، قيد مراجعة، معتمد).
3. ربط الانتقال إلى التخطيط بعد اعتماد القرار فقط.

### المرحلة 4: التخطيط التنفيذي
1. نقل وحدات: النطاق، الميزانية، الخطة، المخاطر إلى ملفات مستقلة.
2. الاحتفاظ بنفس السلوك الحالي أثناء النقل.
3. اعتماد معايير جاهزية قبل فتح المتابعة.

### المرحلة 5: المتابعة والتقارير
1. توحيد منطق المتابعة مع التقارير في دورة تشغيل واحدة.
2. ربط كل تقرير بحالة المرحلة ومشروعه بشكل صريح.
3. ضبط حالات عدم وجود تقرير وعدم وجود مشروع بشكل واضح.

### المرحلة 6: الإغلاق
1. تعريف شروط إغلاق رسمية.
2. إنشاء شاشة إغلاق موحدة بمخرجات نهائية.
3. أرشفة المشروع مع قابلية الرجوع.

### المرحلة 7: إزالة القديم
1. حذف الوحدات القديمة بعد اكتمال كل البدائل.
2. حذف تدريجي وليس دفعة واحدة.
3. تنظيف المسارات والروابط والمكوّنات غير المستخدمة.

## معايير الانتقال بين المراحل
1. اكتمال بيانات المرحلة الحالية.
2. نجاح فحوص الجودة الإلزامية.
3. عدم وجود أخطاء حرجة في المسار.
4. تحقق يدوي لرحلة المستخدم الأساسية.

3. أي انحراف واضح بين السلوك المتوقع والنتيجة الفعلية.

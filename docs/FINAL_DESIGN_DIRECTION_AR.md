# وثيقة اعتماد التوجه التنفيذي للتصميم (نسخة مرجعية)

## 1) الهدف
توحيد اتجاه التصميم والـUX لمشروع **One Minute Strategy** قبل أي تطوير إضافي، مع الالتزام الكامل بأن التغييرات تكون في **طبقة الواجهة فقط** بدون تغيير نطاق المشروع أو منطق التشغيل الحالي.

## 2) المبادئ المعتمدة
1. الصور المرجعية هدفها توسيع الرؤية، وليست للنسخ الحرفي.
2. الحفاظ الكامل على نطاق المشروع الحالي وسير المراحل كما هو.
3. التركيز على وضوح القرار والتنفيذ، لا الزخرفة.
4. بناء واجهة مؤسسية هادئة وقابلة للتوسع.

## 3) التوجه البصري المعتمد
نمط معتمد: **Executive Calm SaaS**

### خصائص النمط
1. تصميم نظيف، احترافي، هادئ.
2. مساحات بيضاء كبيرة وتوزيع مريح.
3. بطاقات بحواف ناعمة (16–20px).
4. حدود خفيفة وتباين واضح للنص.
5. ظلال خفيفة جدًا فقط.
6. ممنوع: التدرجات القوية، التوهج، الظلال الثقيلة، التأثيرات الدرامية.

## 4) دعم العربية وRTL (شرط إلزامي)
1. RTL كامل افتراضيًا في كل الشاشات.
2. ترتيب العناصر والتبويبات من اليمين إلى اليسار.
3. محاذاة عربية صحيحة للنصوص والحالات والبادجات.
4. الالتزام بأنماط تصميم الويب العربي في الملاحة والنماذج.

## 5) الاستجابة للأجهزة (Responsive Strategy)
### Desktop (>= 1200)
1. `Icon Rail + Context Panel + Workspace`.
2. كثافة معلومات أعلى (لوحات + جداول + مؤشرات).

### Tablet (768–1199)
1. أقرب للديسكتوب (معتمد).
2. `Context Panel` قابل للطي + أولوية لمساحة العمل.
3. عرض مبسط للجداول/الأعمدة.

### Mobile (< 768)
1. بدون Sidebar ثابت.
2. `Top Bar + Tabs + Drawer/Bottom Sheet`.
3. بطاقات مختصرة وإجراءات سريعة.

## 6) المعمارية الوظيفية داخل مركز المشاريع
مركز المشاريع يصبح واجهة التشغيل الأساسية بثلاثة تبويبات:
1. **نظرة عامة (Overview)**: مؤشرات تشغيلية، جاهزية، مخاطر، ربحية.
2. **لوحة (Board)**: عرض مهام حسب الحالة.
3. **قائمة (List)**: عرض تفصيلي منظم للمهام.

## 7) السايدبار (استلهام معتمد)
1. اعتماد فكرة `Dual Sidebar`:
   - `Icon Rail` للتنقل الرئيسي.
   - `Context Panel` للتفاصيل حسب القسم.
2. في الجوال يتحول هذا النمط إلى Drawer/Bottom Sheet.
3. لا نسخ لنفس هوية أو تفاصيل الصور المرجعية.

## 8) قواعد الكثافة والمحتوى
1. البطاقة مختصرة وواضحة.
2. التفاصيل المتقدمة تنتقل إلى Modal/Drawer.
3. الجوال أقل كثافة من التابلت والديسكتوب.

## 9) حدود التنفيذ (لا يتم تجاوزها)
1. لا تغيير في API أو العقود أو المنطق.
2. لا تغيير في سير المراحل:
   - `welcome`, `init`, `round1`, `round2`, `dialogue`, `addition`, `done`, `advanced_scope`, `advanced_boq`, `advanced_plan`
3. لا تغيير في منطق البيانات الأساسية:
   - `projectRegistry`, `actionTrackerItems`, `liveRiskItems`, `boqItems`
4. لا كسر للتخزين المحلي الحالي (multi-project + archive/restore + backup/import).

## 10) بيانات المصدر داخل الواجهة الجديدة
يُسمح باستخدام البيانات الحالية فقط لبناء العروض الجديدة:
1. حالة المشاريع (نشط/مؤرشف).
2. المهام التنفيذية وحالاتها.
3. مؤشرات المخاطر.
4. مؤشرات الربحية من BOQ.

## 11) خطة التنفيذ المرحلية (عند بدء التنفيذ)
1. **V1 (تجريبي)**: تحسين هيكل مركز المشاريع فقط مع Responsive كامل + تبويبات Overview/Board/List.
2. **V2**: تحسين تجربة التابلت والجوال وتبسيط الإجراءات.
3. **V3**: تحسينات تشغيلية إضافية (حسب الاعتماد).

## 12) معايير القبول قبل التسليم
1. نجاح `TypeScript` بدون أخطاء.
2. نجاح `Lint` بدون أخطاء.
3. نجاح `Build` للإنتاج.
4. اختبار مسارات أساسية (Desktop + Tablet + Mobile).
5. تأكيد عدم تغيير منطق الأعمال.

## 13) بنود خارج النطاق الحالي (Out of Scope)
1. تغيير نطاق المنتج أو إعادة تعريف وحدات العمل.
2. إضافة كيانات بيانات جديدة بدون اعتماد مسبق.
3. إعادة بناء جذرية للتطبيق خارج مركز المشاريع في النسخة التجريبية.

## 14) قرار اعتماد
هذه الوثيقة تمثل التوجه الرسمي المعتمد للتنفيذ القادم، ويجب الرجوع لها قبل أي تطوير واجهات إضافي.
